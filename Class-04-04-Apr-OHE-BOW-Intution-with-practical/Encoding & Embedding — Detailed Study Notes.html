The Big Picture — Why Do We Encode Text?
The fundamental problem that encoding and embedding solve

Before learning any technique, understand the core problem. Every ML and DL model works with numbers — not words. If you have text data and you want to train a model or make predictions, you first have to convert that text into numerical form.

The machine doesn't understand English. It doesn't understand Hindi. It doesn't understand any human language. It only understands numbers. So whatever text we have — sentences, paragraphs, documents — we have to convert everything into numbers first. That conversion is what encoding and embedding are all about.

— Sunny Savita

The Full Pipeline (Big Picture)

🔄 Text → Numbers → Model → Output

Raw Text (your data) → Encoding / Embedding (convert to numbers) → ML / DL Model (trains and learns) → Predictions / Answers

Without the middle step, no model can work. That's why this chapter is critical — it's the bridge between raw human language and machine intelligence.

Where is This Used in GenAI?

The instructor emphasized that encoding and embedding directly appear in:

  • RAG (Retrieval Augmented Generation) — documents are embedded so similar content can be retrieved
  • Semantic Search — finding similar documents using vector similarity
  • LLM internals — every LLM converts input tokens into embeddings before processing
  • Fine-tuning — the model's embedding layer is updated during training

Historical Timeline

📅 How Techniques Evolved

~2010–2015: Classical era — One-Hot Encoding, Bag of Words, TF-IDF dominated NLP. Used with algorithms like Naive Bayes, Logistic Regression.

~2013: Word2Vec introduced by Google — neural embeddings that capture semantic meaning. Game changer.

~2017–2018: Transformer architecture ("Attention is All You Need" paper). Led to BERT, GPT, and all modern LLMs.

~2019–now: Transformer-based embeddings (SOTA). What powers ChatGPT, Claude, Gemini, etc.

One-Hot Encoding (OHE)
The oldest and simplest text-to-number technique
📖 Definition

One-Hot Encoding represents each word as a binary vector — a list of 0s and 1s — where 1 means the word is present and 0 means the word is absent.

It is called "one-hot" because exactly one position in the vector is "hot" (= 1) at a time, and all others are 0.

Step 2 — Build the Vocabulary

The vocabulary is simply the collection of all unique words from your entire dataset. Repeated words are counted only once.

👉 This vocabulary size becomes the dimension (length) of every vector.

If vocabulary size = 5 → every vector has 5 positions
If vocabulary size = 10,000 → every vector has 10,000 positions

📐 Key Rule: Vocabulary Size = Vector Dimension

If your vocabulary has 5 words, every document is represented as a vector of length 5.

If your vocabulary has 50,000 words (realistic for real data), every vector has 50,000 positions.

Two Levels of OHE Representation

The instructor explained this is a crucial point — there are two different ways to represent data using OHE, used in different contexts:

A) Document-Level Representation

One single vector per entire document/sentence. Used in Classical Machine Learning (Logistic Regression, Naive Bayes) for tasks like text classification.

📄 Document-Level: "People watch movie"

Vocabulary: [people, watch, like, movie, cricket]

Vector = [1, 1, 0, 1, 0]     (Dimension: 1 × 5)

The entire sentence is squashed into one vector of length 5.

B) Word-Level Representation

Each individual word gets its own vector. Only one position is 1; everything else is 0. Used in Deep Learning (RNN, LSTM, GRU). This is actually the classic definition of "one-hot."

🔤 Word-Level: "People watch movie"
people  → [1, 0, 0, 0, 0]   ← position 0 is hot
watch   → [0, 1, 0, 0, 0]   ← position 1 is hot
movie   → [0, 0, 0, 1, 0]   ← position 3 is hot

Combined: 3 rows × 5 columns matrix (3 words, 5 vocab words)

Dimension = number_of_words × vocabulary_size = 3 × 5

RepresentationDimensionUsed InHow It Works
Document-Level1 × vocab_sizeLogistic Regression, Naive BayesEntire sentence → 1 vector
Word-Levelnum_words × vocab_sizeRNN, LSTM, GRUEach word → its own vector

Pros & Cons of One-Hot Encoding

✅ Advantages
  • Extremely easy to understand and implement
  • Simple binary representation (0s and 1s)
  • No model training required — direct mapping
  • Anyone can explain it in an interview
❌ Disadvantages
  • Sparse Matrix — most values are 0, wasting memory
  • High Dimensionality — 50,000 vocab = 50,000-dim vector
  • No Semantic Meaning — "like" and "love" look completely unrelated
  • Out-of-Vocabulary (OOV) — new unseen words can't be represented

The Sparse Matrix Problem — Explained Visually

🚨 Sparse Matrix Problem with Large Vocabulary

Suppose vocabulary size = 10,000 words

Sentence to encode: "People watch movie"

Vector size = 10,000 positions

Positions with 1: only 3 (people, watch, movie)

Positions with 0: 9,997 ← all useless zeros

Is this efficient? No! Is it carrying meaningful information? No!

This is why we don't use OHE in modern NLP for real data.

Out-of-Vocabulary (OOV) Problem — Explained with Example

🔴 OOV Problem — What Happens with New Words

Training Vocabulary: [people, watch, like, movie, cricket]

Now a new document arrives at inference time:

"People enjoy movie"

Problem: "enjoy" does not exist in the vocabulary!

→ We are forced to replace "enjoy" with a 0 (or some special value)

→ The word "enjoy" is completely lost — as if it never existed

This is the Out-of-Vocabulary problem. The model has no way to handle words it has never seen.

2
Bag of Words (BoW)
Moving from presence to frequency — counting how many times a word appears
📖 Definition

Bag of Words represents a document as a vector of word counts — how many times each vocabulary word appears in that document.

It captures word frequency, not just presence. If a word appears 3 times, the count is 3.

The idea behind Bag of Words is — if a word appears more times in a document, it is probably more important. We count the occurrence of each word. That's it. Simple counting.

— Sunny Savita

Key Difference from OHE

💡 OHE vs BoW — The Core Difference

OHE asks: "Is this word present in the document?" → Answer: 1 or 0

BoW asks: "How many times does this word appear?" → Answer: any count (0, 1, 2, 3...)

If a word appears 5 times in a sentence and you use OHE → you write 1. If you use BoW → you write 5.

Step-by-Step Process

  1. 1
    Collect your documents

    Same as before — your input text data (sentences or paragraphs).

  2. 2
    Build the vocabulary

    Same process — extract all unique words. The vocabulary determines vector dimension. Stop words like "a", "the", "and" are typically removed here during preprocessing.

  3. 3
    Count occurrences

    For each document, count how many times each vocabulary word appears. That count becomes the vector value for that position.

Pros & Cons of Bag of Words

✅ Advantages
  • Easy to understand and implement
  • Captures word frequency (more info than OHE)
  • No model training required
  • Works well for classical NLP (text classification, spam detection)
  • Can reflect topic importance via word counts
❌ Disadvantages
  • Ignores word order — "Dog bites man" = "Man bites dog"
  • No semantic understanding — "like" and "love" are unrelated
  • High dimensionality — vocab size = vector size
  • Sparse vectors — mostly zeros for large vocabulary
  • Out-of-Vocabulary problem — new words not handled
  • Overemphasizes frequent words — "movie movie movie" gets high count but no meaning

The Word Order Problem — A Classic Example

⚠️ Critical Flaw: BoW Ignores Word Order

Consider these two very different sentences:

"Dog bites man"    vs    "Man bites dog"

Vocabulary: [dog, bite, man]

BoW vector for BOTH sentences = [1, 1, 1]

Both look identical to the model! The completely opposite meaning is lost.

The OOV Problem in BoW

🔴 Out-of-Vocabulary in Bag of Words

Training Vocabulary: [bite, dog, man]

New sentence at inference: "Lion bites man"

"Lion" → not in vocabulary → count = 0

The word "lion" is completely invisible to the model, even though it changes the entire meaning of the sentence.

Same OOV problem as OHE — a fundamental limitation of all counting-based methods.

The Overemphasis Problem

⚡ Overemphasizing Frequent Words

Sentence: "Movie movie movie"

BoW gives "movie" a count of 3 — high frequency = high importance

But what is the meaning of this sentence? Nothing useful!

The model would treat this as if "movie" is extremely important in this context — even though the sentence has no real meaning.

3
TF-IDF
Term Frequency × Inverse Document Frequency — smarter counting
📖 Full Form & Definition

TF-IDF = Term Frequency × Inverse Document Frequency

TF-IDF measures both the frequency of a word AND its importance across documents. It reduces the weight of very common words (like "the", "and") and increases the weight of informative, rare words.

TF-IDF is not just about counting — it is about asking: how important is this word inside this document? A word that appears everywhere in every document is not very informative. A word that appears often in just one document is very informative for that document. TF-IDF captures exactly this.

— Sunny Savita

The Two Components

ComponentWhat It MeasuresEffect
TF (Term Frequency) How often a word appears in a specific document Words that appear more get higher TF scores
IDF (Inverse Document Frequency) How rare a word is across ALL documents Words common across many docs get lower IDF scores (penalized)
💡 Intuition — Why Multiply TF × IDF?

Common words across all documents (e.g., "the", "is", "and") → High TF, but very Low IDF → TF-IDF score stays LOW → These words are downweighted ✓

Rare, informative words (e.g., "machine learning" in a tech doc) → Decent TF, High IDF → TF-IDF score is HIGH → These words are upweighted ✓

How TF-IDF Fixes BoW's Weaknesses

✓ TF-IDF Improvement Over BoW

BoW problem: "movie movie movie" gets high weight even though it's meaningless repetition.

TF-IDF fix: If "movie" appears in every single document, IDF penalizes it heavily → overall score drops → it's not treated as more important than it really is.

TF-IDF also still suffers from sparse matrices and OOV, but it's significantly better than raw BoW for many NLP tasks.

🎯 Interview Insight — TF-IDF

Q: What is TF-IDF and why is it better than Bag of Words?
TF-IDF stands for Term Frequency × Inverse Document Frequency. While Bag of Words only counts how many times a word appears (which overvalues common but meaningless words), TF-IDF also considers how rare the word is across the entire corpus. Words common to all documents (like "the", "and") get penalized, while words unique to specific documents get rewarded. This gives a better representation of what each document is actually about.
Q: What are the limitations of TF-IDF?
TF-IDF still produces sparse vectors with high dimensionality. It also cannot capture semantic meaning (synonyms look unrelated), does not understand word order or context, and still has the out-of-vocabulary problem for words not seen during training. These limitations drove the development of word embeddings.

Practical 1 — One-Hot Encoding

The instructor used sklearn.preprocessing.OneHotEncoder to demonstrate OHE. Here is the complete code as taught:

Python — One-Hot Encoding 
# Step 1: Import the required library
from sklearn.preprocessing import OneHotEncoder
import numpy as np

# Step 2: Define your documents (dataset)
documents = [
    "people watch movie",
    "people watch cricket",
    "people like movie",
    "people like cricket"
]

# Step 3: Tokenize — split each sentence into individual words
# .lower() converts everything to lowercase (prevents "People" ≠ "people")
# .split() breaks the string on spaces
tokens = [sentence.lower().split() for sentence in documents]
# Result: [['people','watch','movie'], ['people','watch','cricket'], ...]

# Step 4: Convert to 2D word matrix (required format for OneHotEncoder)
# We need each word as a separate row: [['people'], ['watch'], ['movie'], ...]
all_words = [[word] for sentence in tokens for word in sentence]

# Step 5: Create the encoder object
# sparse_output=False → returns a readable NumPy array (not a compressed sparse matrix)
encoder = OneHotEncoder(sparse_output=False)

# Step 6: Fit — encoder learns the vocabulary from your data
encoder.fit(all_words)

# Step 7: View the vocabulary (alphabetically sorted)
print("Vocabulary:", encoder.categories_[0])
# Output: ['cricket' 'like' 'movie' 'people' 'watch']

# Step 8: Transform — convert each sentence to one-hot vectors
for sentence in tokens:
    encoded = encoder.transform([[word] for word in sentence])
    print(f"Sentence: {sentence}")
    print(f"One-Hot Matrix:\n{encoded}\n")

Understanding the Code — What Does Each Step Do?

StepCodeWhat It Does
Tokenize.lower().split()Converts sentence to lowercase and splits into list of words. "People Watch" → ["people", "watch"]
2D Format[[word] for word...]OHE requires each word wrapped in a list: [["people"], ["watch"]]
sparse_output=FalseOneHotEncoder(sparse_output=False)Returns dense NumPy array so we can easily read the output. Default is sparse (compressed format).
fit()encoder.fit(data)Scans all words, learns vocabulary. No transformation yet — just learning.
transform()encoder.transform(data)Applies the learned vocabulary to actually create the one-hot vectors.
fit_transform()encoder.fit_transform(data)Does fit + transform in one single step.
🔑 Understanding fit() vs transform() — The scikit-learn Pattern

fit(data) → "Learn from this data." The encoder reads all the words, builds a vocabulary, validates formats. Nothing is converted yet. Think of it as preparation.

transform(data) → "Now apply what you learned." Takes data and converts it into vectors using the vocabulary created by fit().

fit_transform(data) → Does both in one call. Convenient shortcut when fitting and transforming on the same data.

This pattern (fit/transform/fit_transform) is standard across ALL of scikit-learn — you'll see it in every class, not just OHE.

Practical 2 — Bag of Words

The instructor used sklearn.feature_extraction.text.CountVectorizer for BoW:

Python — Bag of Words (CountVectorizer) 
# Import CountVectorizer — the BoW class in sklearn
from sklearn.feature_extraction.text import CountVectorizer

# Define documents — same dataset as before
documents = [
    "people watch movie",
    "people watch cricket",
    "people like movie",
    "people like cricket"
]

# Create CountVectorizer object
bow = CountVectorizer()

# fit_transform: learn vocabulary AND create word count matrix in one step
# No need for manual tokenization — CountVectorizer handles it internally
bow_matrix = bow.fit_transform(documents)

# View the BoW matrix as an array
print("BoW Matrix:\n", bow_matrix.toarray())

# View the vocabulary (features)
# Note: alphabetically sorted — this determines column order in the matrix
print("\nVocabulary:", bow.get_feature_names_out())

# Example: Transform a completely new sentence
# Words not in vocabulary get count = 0
new_sentence = ["people enjoy movie"]
new_vector = bow.transform(new_sentence)
print("\nNew sentence vector:", new_vector.toarray())
# "enjoy" is OOV — its count will be 0 in the output
💡 Key Difference in How CountVectorizer Works vs OneHotEncoder

With OneHotEncoder, you had to manually tokenize your sentences and create a 2D word array before passing to fit().

With CountVectorizer, you pass the raw sentences directly — it handles tokenization internally. The library was designed this way by its creators.

This is just a design choice by the library authors, not something conceptually different about OHE vs BoW.

Introduction to Embeddings
Why encoding wasn't enough, and what came next

Everything we've learned so far — OHE, BoW, TF-IDF — these are the classical approaches. They work, but they don't understand meaning. Embeddings are completely different. They use neural networks to generate vectors that actually capture the semantic relationship between words. That is the revolution.

— Sunny Savita

Why Embeddings Are Fundamentally Different

Two Types of Embeddings (Upcoming Sessions)

1. Word2Vec

📌 Word2Vec — The Foundation

Introduced by Google researchers (~2013). A neural network trained to predict context words given a target word (or vice versa). After training, each word has a dense vector that captures its meaning.

Famous demonstration: vector("King") - vector("Man") + vector("Woman") ≈ vector("Queen")

This shows that relationships between words are encoded in the vector space — something no classical encoding can do.

Library used: GenSim (requires Python 3.11)

2. Transformer-Based Models (SOTA)

📌 Transformer-Based Embeddings — State of the Art

Built on the Transformer architecture (2017 — "Attention is All You Need" paper). Models like BERT, GPT, Sentence-BERT, etc.

Key advantage: Context-aware embeddings. The word "bank" in "river bank" gets a different vector than "bank" in "bank account" — because context is considered.

This is what powers all modern LLMs — ChatGPT, Claude, Gemini, Llama — and what you'll use in RAG systems.

Why you must learn this: Every RAG system, every semantic search, every vector database (Pinecone, Chroma, Qdrant) uses transformer-based embeddings.

🔮 Where Embeddings Appear in GenAI (Coming in Later Modules)

RAG: Documents are split into chunks → each chunk is embedded → stored in a vector database → at query time, query is embedded → most similar chunks retrieved → fed to LLM for answering.

Semantic Search: Search query and documents are both embedded → cosine similarity finds closest documents.

LLM internals: Every LLM converts input tokens to embeddings as its very first step before any processing.

Interview Preparation
Real questions from the instructor's own GenAI interview experience

🎯 Expected Interview Questions — Encoding & Embedding

Q1: What is One-Hot Encoding? When do you use it?
One-Hot Encoding represents words as binary vectors — 1 if the word is present, 0 if absent. Historically used for text classification with classical ML algorithms like Logistic Regression and Naive Bayes. In deep learning (RNN/LSTM), each word gets its own one-hot vector (word-level representation).
Q2: What is the Out-of-Vocabulary (OOV) problem? How does it affect OHE and BoW?
OOV occurs when a word appears at inference time that was not in the training vocabulary. Since both OHE and BoW can only represent words they've seen, any new word gets a vector of zeros — it's completely invisible. Modern embeddings partially solve this using subword tokenization (e.g., BPE — Byte Pair Encoding used in GPT).
Q3: What is the difference between One-Hot Encoding and Bag of Words?
Both convert text to vectors. OHE represents only presence (0 or 1). BoW represents count/frequency (0, 1, 2, 3...). If a word appears 3 times in a document, OHE gives it a 1, BoW gives it a 3. BoW captures more information but still has no semantic understanding.
Q4: What is TF-IDF? How does it improve upon BoW?
TF-IDF = Term Frequency × Inverse Document Frequency. BoW only counts frequency — TF-IDF also weights that frequency by how rare the word is across all documents. Common words like "the", "is" appear everywhere so they get a very low IDF score, reducing their overall TF-IDF score. This gives more meaningful weights to actually informative words.
Q5: What is a sparse matrix and why is it a problem?
A sparse matrix is one where most values are zero. Both OHE and BoW produce sparse matrices because any given sentence uses only a tiny fraction of the full vocabulary. For a 50,000-word vocabulary, a 3-word sentence produces a vector with 49,997 zeros — wasting memory and computation with no useful information.
Q6: Why did embeddings replace classical encoding methods?
Classical methods (OHE, BoW, TF-IDF) have three fundamental problems: they produce sparse vectors, they have no semantic understanding (synonyms look completely different), and they ignore word order. Embeddings solve all three — they produce dense, low-dimensional vectors where semantically similar words are close in vector space, and transformer-based embeddings capture context/order information.
Q7: What is the role of embedding in RAG systems? (Advanced)
In RAG, every document chunk is converted to an embedding vector and stored in a vector database. When a user asks a question, the question is also embedded. We then find document chunks whose embeddings are most similar to the question embedding (using cosine similarity), retrieve them, and pass them as context to the LLM to generate an accurate answer.
