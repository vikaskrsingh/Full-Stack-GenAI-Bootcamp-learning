Why Are We Moving to SOTA?
What Word2Vec couldn't do — and what transformer-based models solve

We've spent several classes on classical methods: One-Hot Encoding, Bag of Words, TF-IDF, Word2Vec (CBOW + Skip-gram). All of them converted text to numbers, but all had a critical shared flaw — static embeddings. The word "bank" got the same vector whether you were talking about a river bank or a financial bank.

🔑 The Core Upgrade: From Static → Contextual

Word2Vec / classical methods: One fixed vector per word. Context-blind. Only handles single words well.

SOTA (Transformer-based) models: Dynamic vectors that change based on surrounding context. Handle words, sentences, paragraphs, even images and audio — all converted to embeddings with real semantic meaning.

The instructor summarised: "SOTA model is based on the transformer. Using this SOTA embedding model, we can convert any type of data — text, image, audio, video — into meaningful numbers."

📌 Why This Matters for Your Career

For interviews: You must understand transformer architecture, self-attention, and how embeddings are generated inside these models.

For real projects: Every RAG system, semantic search engine, and vector database application uses SOTA embeddings. The instructor said: "RAG is all about the embedding, guys. Everything is embedding inside RAG."

SOTA Embedding Models — The Big Picture
Where the models live, who makes them, and how they work at a high level

Embedding is nothing — it is an intermediate representation of the data. Or I can say, it is an intermediate output of the model. In Word2Vec it came from the hidden layer. In Transformer-based models, it comes after the self-attention mechanism and the feed-forward neural network.

— Sunny Savita

Where Embeddings Come From in a Transformer

In Word2Vec, the embedding was the hidden layer output. In transformer-based models, the pipeline is:

  1. 1
    Tokenisation

    Text is split into tokens (sub-words). Each token gets an initial Word2Vec-style embedding.

  2. 2
    Self-Attention

    Each token's representation is updated by attending to all other tokens in the sequence — this is where context enters. "Bank" near "river" → one representation. "Bank" near "finance" → a different one.

  3. 3
    Feed-Forward Neural Network

    After self-attention, each token goes through a feed-forward layer for further transformation.

  4. 4
    Output = Embedding

    The output of the final transformer layer is the contextual embedding. This is what you get when you call .encode() or embed_query() on any SOTA model.

Sentence Transformers — Open Source Embeddings
The Hugging Face framework for free, local transformer-based embeddings
📖 What Is Sentence Transformer?

Sentence Transformers is a Python framework (made by Hugging Face) for using and training state-of-the-art embedding and re-ranker models. It can compute embeddings from text, images, audio, and video.

Install with: pip install sentence-transformers (already in the course requirements.txt)

Model Used in Class: all-MiniLM-L6-v2

This is one of the most popular open-source sentence embedding models. It maps any sentence/paragraph into a 384-dimensional dense vector. No zeros — all values are meaningful floats (dense vector, not sparse).

Complete Code — Load & Encode

Python — Sentence Transformer (Open Source, Free)
# Install: pip install sentence-transformers
from sentence_transformers import SentenceTransformer

# Load the model (downloads from Hugging Face Hub on first run)
model_name = "all-MiniLM-L6-v2"
open_source_model = SentenceTransformer(model_name)

# ── Single Word ──────────────────────────────────────────
text = "machine"
embedding = open_source_model.encode(text)
print(len(embedding))   # 384
print(embedding.shape)  # (384,)

# ── Sentence ─────────────────────────────────────────────
sentence = "How does machine learning work in neural networks?"
embedding = open_source_model.encode(sentence)
print(embedding.shape)  # (384,) — same dimension!

# ── Full Paragraph ───────────────────────────────────────
paragraph = """Artificial intelligence is transforming every industry.
From healthcare to finance, machine learning models are enabling
new capabilities that were previously impossible."""
embedding = open_source_model.encode(paragraph)
print(embedding.shape)  # (384,) — still 384, regardless of input length!
⭐ Key Insight: Same Dimension Regardless of Input Length

Whether you pass a single word, a sentence, or a full paragraph — the output is always 384 dimensions for this model. This is the power of SOTA models: they compress any amount of text into a fixed-size dense vector that captures its meaning.

This is exactly what makes them useful for RAG: any document chunk gets the same-size vector, so you can compare them with cosine similarity.

Image Embedding with CLIP
How to convert images to vectors using a Vision Transformer
📖 SOTA Models Handle More Than Text

The instructor demonstrated embedding an actual image (his own photo). SOTA models can convert text, images, audio, and video into embeddings — all using the same transformer architecture underneath.

Key Concepts

TermWhat It Means
CLIPContrastive Language-Image Pre-Training — a model trained on both images and their text captions simultaneously. It learns that "a photo of a dog" and a dog image belong in the same vector space.
ViT (Vision Transformer)Vision Transformer — CLIP's image processing backbone. It divides an image into patches and processes each patch like a token in a language transformer. "VIT is called the Vision Transformer."
Pixel ValuesWhen an image is passed to the preprocessor, it's first converted into a grid of numbers (pixel values). These numbers are then fed to the model. An image = a matrix of numbers.
Image EmbeddingThe output of CLIP's vision encoder — a fixed-size float vector that represents what's in the image, in the same space as text embeddings.
Python — Image Embedding with CLIP (Open Source)
from transformers import CLIPModel, CLIPProcessor
from PIL import Image
import torch

# Load the CLIP model (downloads from Hugging Face)
model_name = "openai/clip-vit-base-patch32"
clip_model     = CLIPModel.from_pretrained(model_name)
clip_processor = CLIPProcessor.from_pretrained(model_name)

# Load your image (save a PNG to an 'images/' subfolder)
image = Image.open("images/your_image.png")

# Step 1: Preprocess image → pixel values (numbers)
inputs = clip_processor(images=image, return_tensors="pt")
# inputs contains pixel_values: the image as a tensor of numbers

# Step 2: Pass pixel values to model → get image embedding
# torch.no_grad() = inference only, no training (no gradient tracking)
with torch.no_grad():
    image_embedding = clip_model.get_image_features(**inputs)

print(image_embedding.shape)  # torch.Size([1, 512]) — 512-dim vector
# This vector represents the content of the image in embedding space!
🎯 Why Does This Matter for RAG?

In a multimodal RAG system, you can store image embeddings alongside text embeddings in a vector database. When a user asks "show me photos of a dog," you convert the query text to an embedding, then find the image embeddings closest to it. Same cosine similarity — different data types.

The instructor confirmed: "In multimodal RAG, you can convert image also into the embedding, and on top of it also, we can perform the retrieval operation."

Similarity Search — Three Methods
Cosine similarity, dot product, and Euclidean distance — formulas, ranges, and code

Once we have embeddings, we need a way to measure how similar two embeddings are. Three standard mathematical methods are used:

Cosine Similarity

−1 to +1

+1 = identical direction (very similar)
0 = perpendicular (unrelated)
−1 = opposite direction

Most common in NLP. Measures the angle between vectors, ignoring magnitude.

Dot Product

−∞ to +∞

Higher value = more similar
Sensitive to magnitude (longer vectors score higher even if angle is same)

Fast to compute. Used when vectors are normalised.

Euclidean Distance

0 to +∞

0 = identical vectors (same point)
Higher value = more different (farther apart)

Measures the straight-line distance between two points in vector space.

The Formulas

Cosine Similarity Formula
cosine_sim(A, B) = A · B ÷ (‖A‖ × ‖B‖)
Dot product divided by the product of the magnitudes (norms) of the two vectors
Python — All Three Similarity Methods
import numpy as np

# Two embedding vectors (normally would come from a model)
A = np.array([...])  # embedding of sentence A
B = np.array([...])  # embedding of sentence B

# ── Method 1: Cosine Similarity ───────────────────────────
cos_sim = np.dot(A, B) / (np.linalg.norm(A) * np.linalg.norm(B))
# Range: -1 to 1. Higher = more similar. 0.9 = very similar.

# ── Method 2: Dot Product ────────────────────────────────
dot_product = np.dot(A, B)
# Range: -∞ to +∞. Higher = more similar (if vectors are normalised).

# ── Method 3: Euclidean Distance ─────────────────────────
euclidean_dist = np.linalg.norm(A - B)
# Range: 0 to +∞. LOWER = more similar (smaller distance = closer points).

print(f"Cosine: {cos_sim:.4f}")
print(f"Dot Product: {dot_product:.4f}")
print(f"Euclidean: {euclidean_dist:.4f}")

Full Similarity Search — Query vs. Documents

Python — Semantic Search: Query vs Document Collection
import numpy as np
from langchain_google_genai import GoogleGenerativeAIEmbeddings

# Load model
embed_model = GoogleGenerativeAIEmbeddings(model="models/gemini-embedding-001")

# Query and documents
query = "How to reduce heart disease risk?"
documents = [
    "Eating fiber reduces heart disease risk.",
    "Fruits and vegetables lower cardiovascular problems.",
    "Regular exercise improves heart health.",
    "Stock markets are volatile this quarter.",   # unrelated
]

# Embed query and all documents — USE SAME MODEL!
query_embedding = embed_model.embed_query(query)
doc_embeddings  = embed_model.embed_documents(documents)

# Calculate all three similarity metrics for each document
results = []
for doc, doc_emb in zip(documents, doc_embeddings):
    q = np.array(query_embedding)
    d = np.array(doc_emb)
    
    cosine    = np.dot(q, d) / (np.linalg.norm(q) * np.linalg.norm(d))
    dot_prod  = np.dot(q, d)
    euclidean = np.linalg.norm(q - d)
    
    results.append((doc, cosine, dot_prod, euclidean))

# Sort by cosine similarity (highest first = most relevant)
results.sort(key=lambda x: x[1], reverse=True)
for doc, cos, dot, euc in results:
    print(f"Doc: {doc[:40]}... | Cosine: {cos:.3f} | Dot: {dot:.2f} | Euclidean: {euc:.3f}")

# Expected: heart-disease related docs score high, stock market doc scores low
🎯 Interpreting Results

Cosine similarity: "Eating fiber reduces heart disease risk" vs query → high similarity (~0.85+). "Stock markets are volatile" → low similarity (near 0 or negative). This confirms the model understands meaning, not just word overlap.

For Euclidean: Remember the opposite rule — lower is more similar. The closest (most similar) document will have the smallest Euclidean distance.

Keyword Search vs Semantic Search
Two fundamentally different ways to find relevant documents

I have shown you both methods. One is the keyword search method, and second is the semantic search. Both are used in RAG — the combination of both is called Hybrid Search.

— Sunny Savita

Keyword Search (BM25 style) — How It Works

Counts how many query words appear in each document. Score = number of matching words. Simple, fast, but misses synonyms and semantic meaning entirely.

Python — Keyword Search Implementation
from collections import Counter

query = "How to reduce heart disease risk?"
documents = [
    "Eating fiber reduces heart disease risk.",
    "Fruits and vegetables lower cardiovascular problems.",
    "Regular exercise improves heart health.",
    "Stock markets are volatile this quarter.",
]

# Tokenise query (lowercase + split + remove punctuation)
query_words = query.lower().replace("?", "").split()
# → ['how', 'to', 'reduce', 'heart', 'disease', 'risk']

results = []
for doc in documents:
    # Tokenise document
    doc_words  = doc.lower().replace(".", "").split()
    doc_count  = Counter(doc_words)
    
    # Score = count of query words that appear in document
    score = sum(doc_count[word] for word in query_words)
    results.append((doc, score))

# Sort by score (highest = most keyword overlap)
for doc, score in sorted(results, key=lambda x: x[1], reverse=True):
    print(f"Score: {score} | {doc}")

# Output:
# Score: 2 | Eating fiber reduces heart disease risk.    ← 'heart', 'risk' match
# Score: 1 | Regular exercise improves heart health.     ← 'heart' matches
# Score: 0 | Fruits and vegetables lower cardiovascular. ← no word overlap!
# Score: 0 | Stock markets are volatile this quarter.    ← no overlap

Interview Preparation
Questions covering SOTA embeddings, similarity search, and model selection

🎯 SOTA Embeddings & Similarity Search — Q&A

Q1: What is a SOTA embedding model and how is it different from Word2Vec?
SOTA (State-of-the-Art) embedding models are based on transformer architecture, whereas Word2Vec uses a shallow single-hidden-layer neural network. Key differences: SOTA models produce contextual embeddings (same word gets different vectors based on surrounding context), Word2Vec produces static embeddings. SOTA models handle sentences and paragraphs natively, Word2Vec only handles individual words (requires averaging for sentences). SOTA models can also embed images, audio, and video. Examples: all-MiniLM-L6-v2 (HuggingFace), Gemini embedding-001, OpenAI text-embedding-3-large.
Q2: Where does the embedding come from in a transformer-based model?
In a transformer, the embedding is generated after the self-attention mechanism and the feed-forward neural network. First, tokens get initial Word2Vec-style embeddings. Then self-attention allows each token to attend to all others — this is where contextual understanding is built. The feed-forward layer further transforms the representations. The final output of this pipeline for a sentence is the sentence embedding. This is analogous to Word2Vec's hidden layer output, but far richer because of the multi-head attention mechanism.
Q3: What is cosine similarity and why is it used for semantic search?
Cosine similarity measures the angle between two vectors: cosine_sim(A,B) = (A·B) / (‖A‖ × ‖B‖). Range: -1 to +1. A score of +1 means identical direction (very similar meaning), 0 means unrelated, -1 means opposite. It's preferred for semantic search because it's scale-invariant — it measures the direction of vectors, not their magnitude, so two documents with the same meaning but different lengths still score high similarity.
Q4: What is the difference between keyword search and semantic search?
Keyword search counts exact word matches — a query for "heart disease" won't match "cardiovascular problems" even though they mean the same thing. Score = number of matching words. Semantic search converts both query and documents to embeddings and measures cosine similarity. It understands that "cardiovascular" and "heart disease" are related. In production RAG systems, both are combined (hybrid search): BM25 for keyword retrieval and vector similarity for semantic retrieval, then a re-ranker model selects the best results.
Q5: What are the three similarity methods and their ranges?
Cosine Similarity: range -1 to +1. Higher = more similar. Most common for NLP. Dot Product: range -∞ to +∞. Higher = more similar (best when vectors are normalised). Euclidean Distance: range 0 to +∞. LOWER = more similar (it's a distance, so smaller = closer). Critical rule: always use the SAME embedding model to embed both the query and the documents. Mixing models (Gemini query + OpenAI documents) produces meaningless similarity scores.
Q6: What is CLIP and what is ViT?
CLIP (Contrastive Language-Image Pre-Training) is a transformer-based model trained on image-text pairs. It places images and text in the same embedding space, so you can compare "a photo of a dog" (text) with an actual dog image (pixels) using cosine similarity. ViT (Vision Transformer) is the image encoder inside CLIP — it divides images into patches (like tokens), processes them through transformer layers, and produces an image embedding. Together they enable multimodal RAG systems that can retrieve images using text queries.
Q7: How do you choose which embedding model to use?
For learning and free projects: Hugging Face all-MiniLM-L6-v2 — completely free, 384 dimensions, runs locally. For free production embeddings: Google Gemini embedding-001 — free within API limits, 3072 dimensions. For highest quality production: OpenAI text-embedding-3-large — paid, 3072 dimensions, best benchmark performance. The MTEB (Massive Text Embedding Benchmark) leaderboard on Hugging Face shows objective performance comparisons across all embedding models — use it to choose the best model for your specific use case.
